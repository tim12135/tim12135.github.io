zookeeper分布式锁相关问题

发表于 2025-09-19 分类于 Java Valine：

ZK 节点类型

  • PERSISTENT: 持久节点
  • EPHEMERAL: 临时节点
  • PERSISTENT_SEQUENTIAL: 持久有序节点
  • EPHEMERAL_SEQUENTIAL: 临时有序节点
  • CONTAINER: 容器节点，当该节点下面没有子节点时会被自动删除，zk 3.5.6 后支持
  • PERSISTENT_WITH_TTL: 持久类型的节点可以设置存活时间，当该节点下面没有子节点且超过 TTL 指定时间后会被自动删除，zk
    3.5.6 后支持
  • PERSISTENT_SEQUENTIAL_WITH_TTL: 持久有序类型的节点可以设置存活时间，zk 3.5.6 后支持

ZK 命令

  • ./zkCli.sh -server ip:port
    连接远程 zk 服务

  • create [-s] [-e] [-c] [-t ttl] path [data] [acl]
    创建节点
    -s : 创建有序节点。
    -e : 创建临时节点。
    -c : 创建一个容器节点。
    -t ttl : 创建一个 TTL 节点， -t 时间（单位毫秒）。
    data：节点的数据，可选，如果不使用时，节点数据就为 null。
    acl：访问控制

  • ls [-s] [-w] [-R] path
    使用 ls 命令来查看当前 znode 的子节点
    -w: 监听子节点变化
    -s: 节点状态信息（时间戳、版本号、数据大小等）
    -R: 表示递归的获取

  • get [-s] [-w] path
    获取节点数据信息
    -s: 节点状态信息（时间戳、版本号、数据大小等）
    -w: 监听节点变化

  • delete [-v version] path
    删除某一节点，只能删除无子节点的节点。
    -v: 表示节点版本号

  • deleteall path
    递归的删除某一节点及其子节点

  • stat [-w] path
    查看节点状态信息
    -w: 监听节点变化

Curator ZK 分布式锁

curator:2.11.1, zookeeper:3.4.8

使用InterProcessMutex上锁后，发现会根据提供的 path 创建目录，并在该目录下创建临时有序节点，来实现公平锁的逻辑。

但是在释放锁后，只是将临时有序节点删除了，并没有将 path 目录给删除，会一直留存。

解决办法：升级 zookeeper 驱动 3.5.7，即可支持创建 path 目录时按照 container 模式创建，如上描述，当目录下所有子节点都删除后，会有线程定时扫描删除该节点。

SocketJS

SockJS 网络协议由两部分组成：

  1. 用于识别 SockJS 会话参数的握手
  2. 全双工通信会话

在开始会话之前，SockJS 客户端会向服务器 WS 的 URL+/info来获取服务器的基本信息，包括：

  • websocket: 指定服务器是否需要 WebSocket 传输的属性
  • cookie_needed: 指定服务器是否需要 cookie 支持的属性
  • origins: 允许的来源列表
  • entropy: 随机数生成器的熵源

在 SockJS 开启会话时，客户端会发送请求 /server/session/transport，其中：

  • server: 服务器编号
  • session: 会话编号
  • transport: 传输方式

SockJS 支持的传输方式：

  • WebSocket: 最佳性能，支持全双工通信
  • XHR Streaming: 使用XMLHttpRequest进行流式传输。它使用两个同时的半双工 HTTP 连接（与所有 Comet 技术一样）来模拟全双工
    WebSocket 连接。
  • EventSource: 基于Server-Sent Events的单向通信
  • XHR Long Polling: 使用使用XMLHttpRequest进行长轮询。此传输也使用两个同时的半双工 HTTP 连接来模拟全双工 WebSocket 连接。

客户端在尝试建立连接时，会依次尝试这些传输方式，直到找到一个可用的。

Spring WebSocket

使用 SockJS 时，路径是固定的，无法路径传参，可以使用 query 或者 header 传参。

继承 DefaultHandshakeHandler 重写 determineUser 可以用于识别用户标识。或者通过实现 HandshakeInterceptor 实现 beforeHandshake，用于识别用户标识。

继承 AbstractWebSocketHandler 重写 afterConnectionEstablished 和 afterConnectionClosed，可以用于管理用户会话。

发送消息的时候，通过 redis topic 广播，可以避免服务端多实例，服务端发送消息给客户端时，客户端会话不在当前服务器实例上。

Mysql数据隔离和MVCC

发表于 2024-09-10 更新于 2024-09-11 分类于 Mysql Valine：

并发事务所产生的问题

  1. 脏读
    读到了其他事务未提交的数据。未提交意味着这些数据可能会回滚，读到了不一定最终存在的数据，这就是脏读。
  2. 不可重复读
    在一个事务内，最开始读到的数据和事务结束前任意时刻读到的同一批数据出现不一致的情况。
  3. 幻读
    在一个事务内，相同的查询条件不同时刻读到的结果数量不一致。

隔离级别

  1. 读未提交 READ UNCOMMITTED
    允许事务读取未被其他事务提交的变更。可能会导致脏读、不可重复读、幻读。
  2. 读已提交 READ COMMITTED
    只允许事务读取已经被其他事务提交的变更。可以避免脏读，但可能会导致不可能重复读、幻读。
  3. 可重复读 REPEATABLE READ
    确保事务可以多次从一次字段读取相同的值，在这个事务持续期间，禁止其他事务更新这个字段。可以避免脏读和不可重复读，但可能导致幻读。
  4. 序列化 SERIALIZABLE
    提供严格的事务隔离。事务串行化顺序运行，可以避免所有并发问题，但性能最差。

Mysql 默认隔离级别：REPEATABLE READ，可以通过语句select @@tx_isolation;查询。

MVCC

MVCC 全称 Multi-Version Concurrency Control，即多版本并发控制。其目的是为了尽量避免并发事务读写场景下的上锁。

隐藏列

DB_TRX_ID：更改当前记录的事务 id
DB_ROLL_PTR：回滚指针，指向 undo log 的指针
DB_ROW_ID：innodb 为没有主键和唯一索引的表自动添加的隐藏主键

快照读和当前读

  • 快照读：普通的 select 查询语句（不加锁的 select）就是快照读，快照读是基于 MVCC
  • 当前读：执行 insert、update、delete 以及 select … for update、select … lock in share mode 时，会读取最新版本数据

Undo Log

Undo Log 的目的：

  • 确保事务的原子性。即如果需要回滚，则可以根据 UndoLog 来撤销之前未提交的数据变更。
  • 实现 MVCC。Undo Log 记录了反向操作记录，可得出之前版本的数据，作为数据的旧版本快照，并形成了一条数据版本链，供其他并发事务进行快照读。

Read View

关键字段

  • m_creator_trx_id：创建该 Read View 的事务 ID。
  • m_ids：创建 Read View 时其他未提交的活跃事务 ID 列表。不包括当前事务自己和已提交的事务（正在内存中）。
  • m_low_limit_id：创建 Read View 时最大的事务 ID+1，即下一个将要分配的事务 ID。大于等于这个 ID 的数据版本均不可见。
  • m_up_limit_id：活跃事务列表 m_ids 中最小的事务 ID，如果 m_ids 为空，则 m_low_limit_id 为 m_up_limit_id。小于这个 ID 的数据版本均可见。

Read View 可见性判断

  • DB_TRX_ID == m_creator_trx_id，表示当前事务访问自己修改的记录，因此该版本可以被当前事务访问
  • DB_TRX_ID < m_up_limit_id，表示该版本是在当前事务生成 ReadView 之间提交的，因此该版本可以被当前事务访问
  • DB_TRX_ID >= m_low_limit_id，表示该版本是在当前事务生成 ReadView 之后提交的，因此该版本不能被当前事务访问
  • DB_TRX_ID >= m_up_limit_id && DB_TRX_ID < m_low_limit_id && m_ids 包含 DB_TRX_ID，
    表示创建 ReadView 时生成该版本的事务仍处于活跃状态，因此该版本不能被访问
  • DB_TRX_ID >= m_up_limit_id && DB_TRX_ID < m_low_limit_id && m_ids 不包含 DB_TRX_ID，
    表示创建 ReadView 时生成该版本的事务已提交，因此该版本能访问

RC 和 RR 下的 Read View

  • RC：每次 SELECT 数据前都生成一个 ReadView
  • RR：只在第一次读取数据时生成一个 ReadView，后面会复用第一次生成的

案例

初始化

MYSQL 默认隔离级别为可重复读

场景一

编号 事务 A 事务 B
1 BEGIN;
2 select n from idn where id=1; BEGIN;
3 update idn set n=2 where id=1; select n from idn where id=1;
4 select n from idn where id=1; update idn set n=3 where id=1;
5 COMMIT;
6 select n from idn where id=1;
7 COMMIT;
  • 时间点 3 的时候，事务 A 将 id=1 的记录 n 设置为 2，对于事务 B 来说此时的 ReadView，事务 A 处于活跃中的事务，但是未提交，所以不可见，其查询结果仍是 1
  • 时间点 4 的时候，事务 B 会阻塞，因为事务 A 在时间点 3 执行了 update 语句，对 id=1 的记录上了行锁
  • 时间点 5 的时候，事务 A 提交了，释放了行锁，事务 B 的 update 语句成功执行，id=1 的记录 n 设置为了 3
  • 时间点 6 的时候，事务 B 查询 id=1 的记录，因为最新版本的所属事务就是自己，所以看到的结果 n=3

场景二

编号 事务 A 事务 B 事务 C
1 BEGIN;
2 select n from idn where id=1; BEGIN; BEGIN;
3 update idn set n=2 where id=1;
4 select n from idn where id=1; select n from idn where id=1;
5 COMMIT;
6 select n from idn where id=1; select n from idn where id=1;
7 select n from idn where id=1 for update;
8 COMMIT; COMMIT;
  • 时间点 4 的时候，事务 B 查询 id=1 的记录，因为此时事务 A 处于活跃状态，因此事务 A 的修改，事务 B 不可见
  • 时间点 6 的时候，事务 B 查询 id=1 的记录，因为 mysql 的隔离级别设置为可重复读，所以 ReadView 在时间点 4 的时候已经生成，对于事务 A 的判断是处于活跃中的事务且未提交，所以不可见，查询的结果仍是 1
  • 时间点 6 的时候，事务 C 查询 id=1 的记录，此时生成对应的 ReadView，事务 A 已经提交属于历史版本小于事务 C 的 m_up_limit_id，可见，查询的结果 n=2
  • 时间点 7 的时候，事务 B 查询 id=1 的记录，但是加上了 for update 语句，上了排它锁，属于当前读，读取的是最新版本，查询的结果 n=2

场景三

编号 事务 A 事务 B
1 BEGIN;
2 select n from idn where id=1; BEGIN;
3 update idn set n=n+1 where id=1; select n from idn where id=1;
4 select n from idn where id=1; update idn set n=n+1 where id=1;
5 COMMIT;
6 select n from idn where id=1;
7 COMMIT;
  • 时间点 4 的时候，事务 B 会阻塞，因为事务 A 在时间点 3 执行了 update 语句，对 id=1 的记录上了行锁
  • 时间点 5 的时候，事务 A 提交了，释放了行锁，事务 B 的 update 语句成功执行，id=1 的记录 n=n+1，此时因为是 update 语句，所以是当前读，n 的初始值为事务 A 更新后的值 2，然后再加 1，最终等于 3
  • 时间点 6 的时候，事务 B 查询 id=1 的记录，因为最新版本的所属事务就是自己，所以看到的结果 n=3

什么是 oss？

全称是 Object Storage Service，直译就是对象存储服务。OSS 具有与平台无关的 RESTful API 接口，您可以在任何应用、任何时间、任何地点存储和访问任意类型的数据。

常见的 OSS 供应商

  • 国内：阿里云 OSS，腾讯云 OSS，七牛云 OSS
  • 国外 Amazon AWS，MinIO，其中 MinIO 是开源可以自建

oss 和文件系统的对比

对比项 OSS 文件系统
数据模型 OSS 是一个分布式的对象存储服务，提供的是一个 Key-Value 对形式的对象存储服务。 文件系统是一种典型的树状索引结构。
数据获取 根据 Object 的名称（Key）唯一的获取该 Object 的内容。虽然您可以使用类似 test1/test.jpg 的名称，但是这并不表示 Object 是保存在 test1 目录下。对于 OSS 来说，test1/test.jpg 仅仅只是一个字符串，与 example.jpg 并没有本质的区别。因此不同名称的 Object 之间的访问消耗的资源是类似的。 一个名为 test1/test.jpg 的文件，访问过程需要先访问到 test1 目录，然后再在该目录下查找名为 test.jpg 的文件。
优势 支持海量的用户并发访问。 支持文件的修改，例如修改指定偏移位置的内容、截断文件尾部等。也支持文件夹的操作，例如重命名目录、删除目录、移动目录等非常容易。
劣势 OSS 保存的 Object 不支持修改（追加写 Object 需要调用特定的接口，生成的 Object 也和正常上传的 Object 类型上有差别）。用户即使只需要修改一个字节也需要重新上传整个 Object。OSS 可以通过一些操作来模拟类似文件夹的功能，但是代价非常高。例如重命名目录，如果希望将 test1 目录重命名成 test2，则 OSS 的实际操作是将所有以 test1/开头的 Object 都重新复制成以 test2/开头的 Object，该操作会消耗大量的带宽、存储、计算、时间资源。因此在使用 OSS 时请尽量避免类似的操作。 受限于单个设备的性能。访问越深的目录消耗的资源也越大，操作拥有很多文件的目录也会非常慢。

什么是 bucket？

阿里云中的解释：在上传数据（例如文档、图片、音视频等）到 OSS 之前，您需要在 OSS 所支持的地域中创建一个存储空间（Bucket），然后将无限数量的对象（Object）上传到该 Bucket 中。

我的理解：根文件夹，且所有上传的文件都是平铺在根文件夹中，没有目录结构（实际上有伪目录的概念，是个空的文件，主要用于控制目录下文件的权限）

为什么需要本地文件表？

业务系统中只要有上传文件的功能，就有一定会有用户上传相同的文件名称的文件，此时如果使用本地存储或者 oss 存储，在同一个目录（bucket）下是无法存储相同名称的文件。
因此需要一张本地文件表，记录原始名称和存储名称的映射表。 此外，通过本地文件表，不仅可以统计各业务的文件数、空间消耗，还可以记录文件的 oss 配置，实现文件存储在不同的 oss 服务中。

阿里云 OSS

服务器签名客户端直传 oss 并设置上传回调

https://www.alibabacloud.com/help/zh/oss/use-cases/overview-20?spm=a2c63.p38356.0.0.6b627dfc8Z247J

数据直传至 OSS 是利用 OSS 的表单上传 PostObject 接口。通过客户端先调用应用服务器生成并返回 Post 签名和 Post Policy，再使用这些信息上传到 OSS 服务器。为了将文件信息回写到本地文件表，此时有两种选择：

  1. 如果 OSS 服务支持回调，则可以通过应用服务器返回对应的回调接口、以及指定回调的数据
  2. 如果 OSS 服务不支持回调，则由客户端在上传结束后发起回调，可以复用上述接口

局限性：PostObject 接口不支持分片上传

各种线程泄露问题

发表于 2023-06-19 分类于 Java Valine：

常用排查手段

  • 通过监控工具观察 web 容器所在服务器的状态，一般看 cpu、内存、socket 连接数
  • 通过监控工具观察 jvm 状态，一般看堆内存、gc、io、线程池状态
  • jmap -dump:live,format=b,file=FILE PATH PID dump jvm 内存，观察线程信息和堆内存信息

2023-06-19

事件描述：UAT 环境，进去项目首页，出现部分接口卡死的现象，但是其他接口都能正常请求。

事件原因：dump jvm 内存后，发现都卡死在CompletableFurture#Join这个方法。通过 grafa 监控的线程池状态，当前线程数已经达到最大线程数，且队列中已经排队 2000 多个任务。观察 dump 文件中的线程栈信息，发现有 200 多个容器线程处于等待状态，卡死在CompletableFurture#Join这个方法，同时线程池中的线程也处于处于等待状态，卡死在CompletableFurture#Join这个方法。根据线程池的线程栈调用信息，发现都是在某个业务类的方法进来，该业务类方法也开启了线程调用。

根本原因：总结下来就是，使用了同一个线程池，开启线程异步执行任务 A，任务 A 中又开启了线程异步执行任务 B，在并发场景高的情况下，可能会出现线程池中得到线程执行的都是任务 A，所有的任务 A 都执行到开启线程执行任务 B的时间点，但是由于活跃线程都被任务 A 占用了，所以任务 B 都在线程池队列中等待，而任务 A 又都在等待任务 B 返回结果，最终导致线程得不到释放而耗尽

CAPA：

  1. 在同一个线程池中开启的线程在执行过程中不允许再开启该线程池中的其他线程去执行任务
  2. 在调用外部服务的时候，需要配置超时熔断机制，避免卡死后带来的雪崩

2023-01-28

事件描述：生产环境，首页白页，所有请求后端接口卡死

事件原因：通过 grafa 观察后端服务器状态，发现 socket 连接数搞到 8k，web 容器的线程数也达到 500 多，没有得到释放。

根本原因：dump jvm 内存后，发现线程栈信息，都卡死在CountDownLatch#await上，根据堆栈提示的类和代码行数，找到发现在使用多线程调用时，线程执行过程中抛出了运行时异常，countDown 方法没有放在 finally 中，得不到执行，且 await 方法没有设置超时时间，导致主线程一直阻塞在 await 方法，最终 web 容器的线程池耗尽而卡死。

CAPA：

  1. 所用用到闭锁的地方，都需要将释放锁的逻辑放到 finally 中，且 await 都需要设置超时时间
  2. 代码提交的时候用 sonarlint 扫描，避免这些 block 问题
  3. 配置 actuator 监控线程池的使用状况，配置预警和告警通知，及时发现线程池状态不健康等问题

2022-UN

事件描述：生产环境，首页白页，大部分请求后端接口卡死，过一段时间全部接口全部卡死

事件原因：通过 grafa 观察后端服务器状态，发现 socket 连接数搞到 8k，web 容器的线程数也达到 500 多，没有得到释放。

根本原因：微服务调用存在循环调用的问题。服务 A 调用服务 B，服务 B 又调用了服务 A。并发高时，循环调用导致了服务 A 的 web 容器线程数一下子上来，从而导致了服务 B 想要调用服务 A 得排队，继而导致服务 A 线程得不到释放，外部请求都在队列中排队，最终卡死。

CAPA：

  1. 微服务间避免循环调用，划清微服务的功能界限，只能单向调用
  2. 微服务请求一定要设置超时时间和熔断机制，避免雪崩

UML 意义

很多时候，客户跟产品经理交流，产品经理再跟开发交流，开发再和其他开发交流。在沟通的过程中，很多需求可能变了味，不再是客户最开始所描述的。其根本原因在于每个人的思维方式和理解能力是不同的。UML 的意义就是它定下了一套建模的标准，把抽象的文字转换成了更形象的图形。你拿一个时序图或者活动图比你巴拉巴拉讲半天更能让其他人明白整个业务流程的流转交互。

依稀记得大学的老师跟我们说过如果能熟练掌握 UML 建模，那么你应该可以很轻松的敲出代码。

现在想想的确如此，客户的需求用用例图来表示明确功能点，复杂交互可以用活动图、时序图来表示，对象的状态变更可以用状态机图来表示。包图、类图可以化解多人开发时信息孤岛、命名冲突、不规范等问题，组件图可以有助于了解组件、包或者三方库之间的依赖关系，部署图可以让运维架构师们清晰的了解各软件服务所需的物理资源以及分布情况。

Spring Security学习笔记

发表于 2023-05-16 更新于 2023-05-25 分类于 Java ， Spring Valine：

shiro、spring security 这两个安全框架一直有听说，但一直没有使用过。从事过的几家公司，也都是自建安全框架。近期公司在搞数据权限设计，正好趁此机会学习下 spring security。

对安全框架的认识

  1. 登录验证

通过用户名、密码或者手机号、验证码等方式，让系统知道你是谁的一个步骤，并且认证通过后会发给你一个凭证（token、sessionId 等），接下来都是通过这个凭证来访问。

  1. 权限校验

你拿着凭证，只能访问管理员赋予你权限访问的地址。这里即包括在前端页面（web、app 等）可以看到的菜单、按钮等页面信息，也包括每次动作请求的后端接口。如果没有权限对于前端页面来说你就看不到相关信息，对于后端接口来说你请求会返回无权限的标识，例如 http 的 403 状态。

  1. 如何实现
  • a. 实现一个登录接口，校验通过后，生成凭证，并将用户的信息和权限缓存起来。
  • b. 实现一个 filter，对每个请求判断是否有凭证，如果没有凭证则返回 401 信息，如果有凭证则拿出用户信息和权限进行校验，如果凭证错误，则返回 401 信息，如果权限不足则返回 403，有权限则允许继续处理。

spring security

名词解释

  • Authentication
    认证，即用于确认用户身份的过程，一般来说就是校验用户名和密码。
  • Authorization
    授权，即确定用户是否有权访问资源或执行特定操作的过程。授权通常是访问控制的第二步，因为它确保只有经过授权的用户才能访问系统中的资源。
  • Protection Against Exploits
    防御攻击漏洞，主要是针对 CSRF(跨站点请求伪造)、Security HTTP Response Headers(安全响应头)
  • Principal
    在计算机中一般代指一个用户、一个实体。在 Authentication 过程中，一般就是指用户的用户名或者其他什么唯一标识这个用户的信息。
  • Credentials
    指身份验证信息，一般就是密码、数字证书、生物识别信息等。

security 的几个核心过滤器

  • DelegatingFilterProxy

    过滤器代理，其可以将请求委托给一个 Spring 管理的 Filter Bean。通过配置targetBeanName来指定具体的一个实现 Filter 接口的 Bean。这里的意义在于可以延迟加载这个 Bean，且这个 Bean 由 Spring 上下文管理。

  • FilterChainProxy

    security 提供的过滤器链代理，其可以将请求委托给一组过滤器链来处理，这些过滤器链可以根据请求的 URL 路径进行匹配，并按照一定的顺序依次处理请求。在配置 FilterChainProxy 时，可以使用多个 SecurityFilterChain 实例来定义不同的过滤器链，并将其组合成一个完整的过滤器链。每个 SecurityFilterChain 实例都包含了一个或多个过滤器，这些过滤器可以处理特定的 URL 路径。

  • SecurityFilterChain

    security 提供的过滤器链，它定义了一组过滤器，这些过滤器可以处理特定的 URL 路径。

  • Security Filters

    我列出一些使用到的的 Security Filter，如下：

    • CorsFilter，处理跨域请求
    • CsrfFilter，处理 Csrf
    • LogoutFilter，登出
    • UsernamePasswordAuthenticationFilter，用户名密码登入，这里还有各种登入过滤器（OAuth2.0，SAML2.0，CAS 等）
    • AnonymousAuthenticationFilter，未登录用户给一个匿名用户身份
    • ExceptionTranslationFilter，用于处理认证异常和访问被拒异常
    • FilterSecurityInterceptor，用于访问权限判定

雪花算法自动生成workId

发表于 2023-04-26 更新于 2023-05-25 分类于 算法 Valine：

Twitter 的雪花算法大部分人应该都知道，用于生成有序的数值类型的 id。算法的原理也很简单，通过划分了 64 位比特：1 位符号位固定为 0 表示正数，41 位时间戳，5 位数据中心 id，5 位机器 id，12 位有序序号。其中 5 位数据中心 id 和 5 位机器 id，接下来我将合并成 10 位 workid 来做讨论，含义是一样的，总共可以分配 1024 台机器用于生成相同业务的主键。

为啥要区分 workid？

同一种业务他的主键一定是唯一的。如商品服务，则商品表内的主键肯定是不能重复的。但如果发布多个商品服务实例，此时 id 如果是有商品服务自己生成，则不区分 workid，那么在相同毫秒下如果出现并发请求且请求到了不同的实例，理论上是会出现 id 重复的。虽然概率很小，但是在一些集中时间高并发的业务是可能出现的，我们还是应该要劲量避免这种问题。

怎么解决

  1. 手动配置
    单机情况下，我们一般在实现该算法的时候，可能就直接指定了 workid 为一个固定的整数（0~1023）。或者集群环境下，有一台或多台服务专门提供生成主键的服务，此时也可以通过手动分配（硬编码或者配置文件）的方式来指定 workid。

    但是随着容器化自动部署，共用配置中心，相同的服务用的是同一套配置中心，很难通过手动的配置方式来指定 workid。即使可以做到，对于后期的运维也是很麻烦的事情。

  2. 美团 leaf
    市面上，当前有美团的开源项目 leaf，将业务服务的 ip+port 添加到 zookeeper 上，通过 zookeeper 的分布式一致性以及它的持久性节点，来避免抢占相同的 workid，同时 leaf 还通过定时任务上传业务服务的时间信息，来避免时钟回拨的问题，可以仔细阅读下代码，这块代码还是很好理解：https://github.com/Meituan-Dianping/Leaf

    但是我司在 k8s 上部署服务的时候，并没有指定 ip，每次发布 ip 都是动态分配的，也就意味着相同业务实例它的 workid 会随着发布次数的变多，就会超过 1024 个，有人也提了 issue：https://github.com/Meituan-Dianping/Leaf/issues/84，有热心市民也给出了解决方案：https://github.com/NotFound9/Leaf。

自己实现

基于 zookeeper 的临时节点的特性，即客户端断开连接，就会删除节点。可以实现一个自动生成不重复的 workId 的功能。

  1. 参考美团的 snowflake 算法，我把 workId 给暴露出去，同时提供了 init 和 destroy 方法，用于容器内的初始化和销毁，主要用于 zk 客户端
  2. 创建了一个 ZKSnowflake 继承 Snowflake
  3. 在 init 方法中，通过分布式锁先锁住对应业务类型的节点，用于该业务类型多实例发布的时候避免抢占。
  4. 遍历该业务节点下的所有节点，然后再 0~1023 这个范围内遍历，选出还没有被使用的节点，将这个下标作为 workid，并创建到 zk 中作为一个临时节点。
  5. zk 客户端增加监听，对重连的事件，进行重新获取 workid，期间内不允许生成 id。（该类情况可以根据业务的实际场景来判断，其实如果都是指定时间发布的，后续不做修改也可以，只要在按批发布的时候区分好 workid 即可）

总结

相比于美团的，自己实现的方法比较简单，但是没法解决时钟回拨的问题（因为无法知道该服务实例上一次生成 id 的时间戳），也可能会遇到 zk 丢失节点（临时节点）等一些问题。
