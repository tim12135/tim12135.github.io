2022-03-03招聘总结

发表于 2022-03-07 分类于 工作总结 阅读次数： Valine：

画像（希望招聘到的人）

  1. 熟练使用 Spring，aop（spring 的动态代理方式：jdk(利用拦截器(拦截器必须实现 InvocationHanlder)加上反射机制生成一个实现代理接口的匿名类)、cglib(利用 ASM 开源包，对代理对象类的 class 文件加载进来，通过修改其字节码生成子类来处理)）、事件通知（继承 ApplicationEvent 且监听器实现 ApplicationListener 或者@EventListener、@TransactionalEventListener）、事务（传播机制：required、required_new）、同一对象方法间调用事务失效
  2. 熟练使用 Mybatis，对 Mybatis 的一级缓存（session）、二级缓存(namespace)有所了解，对# 、$知悉，接口绑定方式：注解、xml
  3. 熟练掌握 Mysql，事务的隔离级别（读未提交、读已提交、可重复读、串行），索引类型（B+Tree、hash（=、in、≠）），锁的类型（全局、表、行（键、间隙锁、临键锁）），死锁（原因：（多个进程持有资源不释放，并且需求别的进程占用的资源，而导致死锁），避免：（事务颗粒度小，按顺序访问某一对象））
  4. 数量掌握 SpringCloud，服务注册发现，负载均衡，会使用服务限流、熔断、降级，hystrix（熔断策略：线程池和信号量）、sentinel
  5. 熟练掌握 RabbitMQ、Kafka 等消息中间件，消息队列有什么优缺点，交换器 exchange 的 4 中类型（fanout、direct（默认）、topic（routeKey）、headers），确保消息的有序消费。
  6. kafka broker、topic、partition，kafka 的消费模式（推、拉（是配置消费速率不同的消费者，没有数据时设置 timeout 控制获取不到数据时的查询等待时间）），怎么能确保消息的有序消费。
  7. 熟练掌握 Redis，基本类型（String(opsForValue),Hash(opsForHash),List,Set,ZSet），缓存击穿(热点数据过期，通过互斥更新，随机退避，热点 key 同时失效则加随机时间)、缓存穿透（不存在的 id 频繁请求，通过记录空表及或者 BloomFilter）、缓存雪崩（服务降级，主从模式和集群模式），
  8. 熟练掌握 POI，对 word、excel(XSSFWorkbook)操作，知道 2003（65536 行，256 列）和 2007（1048576 行，16384 列）的区别
  9. 对字符集（UTF-8、GBK、UTF-8 BOM(头上有 U+FEFF)）了解区别
  10. 线程池，CountDownLatch ，CompletableFuture
  11. 熟练使用 git 版本管理，会使用 skywalking 追踪日志
  12. 能单独完成业务需求，从数据结构设计到代码实现以及代码自测和部署

准备的问题

  1. 简单介绍下自己之前做过的项目以及担任的角色和使用的技术栈
  2. 在项目中有没有碰到什么技术上的难题，怎么解决的
  3. spring 怎么解决循环依赖的，知道三级缓存，一级缓存日常的 bean，二级属性还没有注入，三级对象工厂
  4. spring 的事务实现知道吗，同一个对象内的方法级别调用，事务会生效吗？
  5. spring 的事务传播机制，主要辨识下 Required 和 RequiredNew 的区别
  6. spring 的事件通知知道吗，讲下怎么使用，继承 ApplicationEvent 实现 ApplicationListener，或者直接加注解@EventListener、@TransactionalEventListener
  7. redis 常用的几种数据类型，以及什么场景使用 redis
  8. 知道 redis 缓存穿透吗，解决方案
  9. mysql 的索引类型有了解过吗，能讲讲索引优化吗，mysql 的事务隔离级别了解吗（读未提交、读已提交、可重复读、串行）
  10. 死锁的原因，解决方案
  11. (rocketMq)ONS 和 Kafka 有什么大的区别，各自使用场景？不依赖 zk，避免了主从选举，分区是逻辑概念上的分区，物理文件还是同一个
  12. 微服务的框架 spring cloud 接触过吗，了解服务限流、熔断、降级这些吗

结论

总体来讲 T4~T4+

  1. 对数据库，mysql 的使用应该较弱，对索引的认知不够
  2. 对微服务的调用、熔断这块不是很清晰
  3. 对消息中间件了解比较透彻
  4. 对 redis 的使用可能比较片面
  5. 对自定义表单的概念比较模糊，倾向于通过文档型数据库解决
  6. 业务分析能力应该不错
  7. 表述比较积极
  8. 倾向于工具使用，对原理不够深入

总结

  1. 面试前需要根据对方的简历和预期的画像，提炼出想要问的问题
  2. 提问需要精准有技巧，提问的目的是为了了解面试者的能力，用于判定是否能满足岗位需求
  3. 提问后没有得到预期的答复，在当时不了解的情况下，应该记录下当时面试者的回答，事后再研究其回答的优劣
  4. 重点关注的问题，如果面试者不了解，不一定代表对方不懂，需要稍加引导，关注对方的思路
  5. 已经得到答案之后，不管对错，都不需要再展开这方面的讨论而浪费时间
  6. 在关心技术能力之外，还得观察对方的表述、对方的性格，事后可以做一个比较全面的评价
